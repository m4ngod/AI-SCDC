# AI Company Console Phase 1 Planner Approval Design

## Goal

Build the Planner-only product loop for **AI Software Company Desktop Console**: a user submits a project goal, a deterministic Planner creates one or more structured `TaskSpec` drafts, the user approves or rejects the batch, and approved drafts become persisted tasks visible in the desktop task board.

## Confirmed Decisions

- Phase 1 scope is **Planner Approval Loop** only.
- Planner implementation is deterministic `FakePlanner` for now.
- The system keeps a clear Planner/LLM interface so Phase 2 can replace the fake implementation with a real model-backed planner.
- Planner output is one or more `TaskSpec` drafts, not a single fixed task and not a full project plan tree.
- Approval is batch-level only: approve or reject all drafts from one planner run.
- Dispatch, mock execution, runner behavior, reviewer behavior, model routing, BYOK, billing, and real LLM calls remain out of scope.
- The implementation should use first-class persistence objects instead of hiding approval state inside conversation message JSON.

## Scope

Phase 1 extends the Phase 0 monorepo with a complete planning and approval workflow.

Phase 1 includes:

- `PlannerRun`: one planning request created from a user goal within a project and optional conversation.
- `PlannerTaskDraft`: one structured task draft generated by a planner run.
- `Approval`: one batch decision for a planner run.
- `FakePlanner`: deterministic planner service that maps a user goal to valid `TaskSpec` drafts.
- API endpoints for creating planner runs, reading runs and drafts, approving a run, and rejecting a run.
- Batch approval behavior that converts drafts into normal persisted `Task` rows and `task_created` events.
- Desktop UI that shows generated drafts in the main thread and lets the user approve or reject the batch.
- Tests covering API behavior, planner determinism, protocol-compatible fields, desktop draft preview, and approval flow.

Phase 1 excludes:

- Real model calls.
- Model provider credentials.
- Model routing.
- Agent dispatch after approval.
- Worker execution.
- Diff generation.
- Per-draft editing.
- Per-draft approval.
- Draft version history.
- Production authentication.
- Billing or usage ledger writes.

## Domain Model

### PlannerRun

`PlannerRun` represents one attempt to turn a user goal into task drafts.

Required fields:

- `id`
- `project_id`
- `conversation_id`
- `goal`
- `status`
- `planner_kind`
- `draft_count`
- `created_by`
- `created_at`
- `updated_at`

Status values:

- `DRAFTED`: drafts were generated and are waiting for user decision.
- `APPROVED`: the user approved the batch and tasks were created.
- `REJECTED`: the user rejected the batch and no tasks were created.

`planner_kind` is `fake` in Phase 1. It exists so future implementations can distinguish `fake`, `openai_compatible`, or other planner backends without changing the public API.

### PlannerTaskDraft

`PlannerTaskDraft` stores a structured draft produced by a planner run. It must align with `packages/agent-protocol/schemas/task-spec.schema.json`.

Required fields:

- `id`
- `planner_run_id`
- `sequence`
- `title`
- `role_required`
- `objective`
- `acceptance_criteria`
- `allowed_paths`
- `required_tests`
- `risk_level`
- `created_at`

`role_required` must use the existing Agent role enum:

- `planner`
- `frontend`
- `backend`
- `reviewer`
- `debugger`
- `security`
- `product`
- `documentation`

`risk_level` must be one of:

- `low`
- `medium`
- `high`

### Approval

`Approval` records the user decision for a planner run.

Required fields:

- `id`
- `workspace_id`
- `project_id`
- `planner_run_id`
- `action_type`
- `risk_level`
- `reason`
- `status`
- `decided_by`
- `decided_at`
- `created_at`

Phase 1 supports only planner-run approvals:

- `action_type`: `approve_planner_run`
- `status`: `approved` or `rejected`

The API must reject a second approval or rejection for the same `PlannerRun`.

## Fake Planner Behavior

The fake planner is deterministic and has no network dependency. Given the same project and goal, it returns the same ordered draft shape.

For Phase 1, the fake planner should produce two drafts by default:

1. A `frontend` draft focused on the user-facing flow or desktop UI.
2. A `backend` draft focused on the API, persistence, and tests needed to support that flow.

The generated drafts should include the user's goal in the title or objective so the preview feels connected to the request. The fake planner must not produce invalid protocol fields, unsupported roles, or unsupported risk levels.

The fake planner should expose a narrow interface:

```python
class PlannerService(Protocol):
    def plan(self, project_id: str, goal: str) -> list[TaskSpecDraft]:
        ...
```

`TaskSpecDraft` is an internal value object or Pydantic model, not a database entity. The repository layer persists it as `PlannerTaskDraft`.

## API Design

### Create Planner Run

```text
POST /projects/{project_id}/planner-runs
```

Request:

```json
{
  "goal": "Build a settings page for model routes.",
  "conversation_id": "conversation_abc"
}
```

Response:

```json
{
  "id": "planner_run_abc",
  "project_id": "project_abc",
  "conversation_id": "conversation_abc",
  "goal": "Build a settings page for model routes.",
  "status": "DRAFTED",
  "planner_kind": "fake",
  "draft_count": 2,
  "drafts": [
    {
      "id": "planner_draft_1",
      "sequence": 1,
      "title": "Design model route settings UI",
      "role_required": "frontend",
      "objective": "Create the desktop UI for reviewing and configuring model routes.",
      "acceptance_criteria": [
        "The user can inspect generated settings UI copy.",
        "The UI can show validation and empty states."
      ],
      "allowed_paths": ["apps/desktop/**"],
      "required_tests": ["Desktop planner approval flow renders draft preview"],
      "risk_level": "medium"
    }
  ]
}
```

Behavior:

- Returns 404 if the project does not exist.
- Returns 400 if `conversation_id` belongs to a different project.
- Creates one `PlannerRun` and at least one `PlannerTaskDraft`.
- Does not create normal `Task` rows.

### Get Planner Run

```text
GET /planner-runs/{planner_run_id}
```

Returns the run and ordered drafts.

Behavior:

- Returns 404 if the run does not exist.

### Approve Planner Run

```text
POST /planner-runs/{planner_run_id}/approve
```

Response:

```json
{
  "planner_run_id": "planner_run_abc",
  "approval_id": "approval_abc",
  "status": "APPROVED",
  "created_tasks": [
    {
      "id": "task_abc",
      "title": "Design model route settings UI",
      "status": "CREATED",
      "role_required": "frontend"
    }
  ]
}
```

Behavior:

- Returns 404 if the run does not exist.
- Returns 400 if the run has already been approved or rejected.
- Creates one `Approval` with `status = approved`.
- Creates one normal `Task` per draft.
- Maps `PlannerTaskDraft.objective` into `Task.description`.
- Maps draft `acceptance_criteria`, `role_required`, and `risk_level` into the created task.
- Creates `task_created` events for each task.
- Updates the run status to `APPROVED`.

### Reject Planner Run

```text
POST /planner-runs/{planner_run_id}/reject
```

Request:

```json
{
  "reason": "The scope is too broad for this project."
}
```

Behavior:

- Returns 404 if the run does not exist.
- Returns 400 if the run has already been approved or rejected.
- Creates one `Approval` with `status = rejected`.
- Updates the run status to `REJECTED`.
- Does not create normal `Task` rows.

## Desktop Design

The desktop keeps the Phase 0 shell structure:

- Left navigation stays unchanged.
- Main thread owns goal submission and Planner draft preview.
- Right context panel owns task board and operational context.

The goal input flow changes from direct task creation to planning:

1. User enters a goal.
2. Desktop calls `POST /projects/{project_id}/planner-runs`.
3. Main thread renders the generated draft list.
4. User can approve or reject the entire batch.
5. On approve, desktop calls `POST /planner-runs/{id}/approve`.
6. Created tasks are prepended into the right-panel task board.
7. On reject, desktop marks the draft preview rejected and does not update the task board.

In mock desktop mode, the client uses deterministic fake planner responses without a backend. In API-backed mode, it calls the FastAPI planner endpoints.

The UI should remain dense and work-focused. The draft preview should show:

- Draft title.
- Role.
- Risk level.
- Objective.
- Acceptance criteria.
- Allowed paths.
- Required tests.

Phase 1 does not include inline editing of draft fields.

## Error Handling

The API must handle:

- Missing project: 404.
- Cross-project conversation on planner run creation: 400.
- Missing planner run: 404.
- Repeated approve or reject: 400.
- Approval of a rejected run: 400.
- Rejection of an approved run: 400.
- Invalid enum values in persisted or request data: 422 for request validation, test failure for fake planner output.

The desktop must display inline errors for:

- Planner run creation failure.
- Approval failure.
- Rejection failure.
- Initial task loading failure, preserving the Phase 0 behavior.

## Testing Strategy

### API Tests

Add tests for:

- Creating a planner run creates ordered drafts and no normal tasks.
- Creating a planner run with a missing project returns 404.
- Creating a planner run with a cross-project conversation returns 400.
- Getting a planner run returns drafts in sequence order.
- Approving a planner run creates tasks and task events.
- Rejecting a planner run creates no tasks.
- Repeating approval or rejection fails with 400.
- Draft role and risk values align with `agent-protocol`.

### Planner Tests

Add tests for:

- Fake planner returns deterministic output for the same goal.
- Fake planner returns at least one draft.
- Fake planner returns protocol-compatible roles and risk levels.
- Fake planner includes the user goal in generated draft content.

### Desktop Tests

Add tests for:

- Fake client creates planner drafts from a goal.
- App renders planner draft preview in the main thread.
- Approving drafts calls the client and adds created tasks to the task board.
- Rejecting drafts does not add tasks.
- Planner creation and approval errors render inline.
- HTTP client maps API planner-run responses to desktop view models.

## Success Criteria

Phase 1 is complete when:

- A user can submit a goal and see multiple structured TaskSpec drafts.
- A user can approve the batch and see created tasks in the task board.
- A user can reject the batch and create no tasks.
- Planner drafts are persisted separately from normal tasks.
- Approval decisions are persisted separately from conversation messages.
- Fake planner output validates against the supported Agent role and risk enums.
- The whole repository still passes `pnpm test`, `pnpm typecheck`, and Python tests.

## Out-of-Scope Deferral Notes

Phase 2 can replace `FakePlanner` with a real model-backed planner through the same planner interface, then add provider credentials, model routes, BYOK credential scaffolding, and usage logging.

Phase 3 can use approved tasks as input to Local Runner worktree and patch generation workflows.
