# AI Company Console Phase 2 Model Routing and BYOK Design

## Goal

Build the backend-first model routing and BYOK foundation for **AI Software Company Desktop Console**. Phase 2 gives the control plane durable records for model providers, user-owned credentials, agent-role model routes, route resolution, and append-only usage logging while preserving the Phase 1 fake planner loop.

## Confirmed Decisions

- Phase 2 scope is **Control-plane Foundation**.
- Phase 2 is backend-first. It does not include a desktop model settings page.
- Phase 2 does not make real OpenAI, DeepSeek, Anthropic, Gemini, or other provider network calls.
- Phase 2 does not replace `FakePlanner` with a real model-backed planner.
- `FakePlanner` remains the default planner implementation.
- The API can configure a planner route, but route resolution is metadata-only in this phase.
- DeepSeek is represented as an OpenAI-compatible provider configuration, not as hard-coded model behavior.
- BYOK credentials are write-only through the API. Responses and OpenAPI schemas must not expose raw secrets or encrypted secret storage fields.
- Usage logging is append-only. Phase 2 has create/list behavior only, with no update or delete route.

## Scope

Phase 2 extends the Phase 0 and Phase 1 control-plane backend with the minimum durable model-routing substrate needed for later real model execution.

Phase 2 includes:

- Persistent `ModelProvider` records for provider metadata and OpenAI-compatible configuration.
- Persistent `ModelCredential` records for BYOK metadata and encrypted-secret placeholders.
- Persistent `ModelRoute` records that bind an `agent_role` to provider, model name, optional credential, and fallback model names.
- Persistent `UsageLedgerEntry` records for append-only model usage accounting.
- Service-layer functions for creating/listing providers, creating/listing/deleting credentials, creating/updating/listing routes, resolving a route by agent role, and appending/listing usage ledger entries.
- API endpoints for model providers, credentials, routes, route resolution, and usage ledger queries.
- A development `SecretVault` abstraction that masks and stores credentials without ever returning raw secret values.
- Tests proving secrets are write-only, route resolution falls back to a deterministic fake route, routes reject invalid references, and usage ledger entries are append-only.
- Documentation updates describing Phase 2 boundaries.

Phase 2 excludes:

- Real LLM provider HTTP calls.
- Real cloud KMS, OS keychain, or production secret manager integration.
- Desktop model settings UI.
- Real planner output generated by models.
- Streaming model responses.
- Model pricing calculation beyond an `amount_cents` placeholder.
- Billing collection, invoices, subscriptions, credit wallet mutation, or payment provider integration.
- Per-organization production auth and RBAC.
- Provider health checks that call external networks.

## Domain Model

### ModelProvider

`ModelProvider` stores a model supplier configuration available inside the development workspace.

Required fields:

- `id`
- `workspace_id`
- `name`
- `provider_type`
- `base_url`
- `default_headers`
- `status`
- `created_at`
- `updated_at`

`name` is unique per workspace.

Supported `provider_type` values:

- `fake`
- `openai_compatible`
- `deepseek`

`deepseek` is a provider type so the UI and API can show intent clearly, but the runtime contract should treat it as OpenAI-compatible metadata. Phase 2 does not hard-code a DeepSeek model list.

Supported `status` values:

- `active`
- `disabled`

`default_headers` stores non-secret headers only. Secret-bearing headers belong in `ModelCredential`.

### ModelCredential

`ModelCredential` stores BYOK credential metadata and a development encrypted-secret placeholder.

Required fields:

- `id`
- `workspace_id`
- `provider_id`
- `display_name`
- `secret_last4`
- `encrypted_secret`
- `status`
- `created_at`
- `updated_at`

Supported `status` values:

- `active`
- `deleted`

The API accepts `secret_value` only on create. `secret_value` is never persisted directly, never returned by any endpoint, and never included in response schemas.

The API also never returns `encrypted_secret`. That field is internal storage only.

Deleting a credential is a soft delete. It sets `status = deleted` and updates `updated_at`. Routes that reference a deleted credential remain stored, but route resolution must mark the resolved route unavailable.

### ModelRoute

`ModelRoute` binds an agent role to model-selection metadata.

Required fields:

- `id`
- `workspace_id`
- `agent_role`
- `provider_id`
- `credential_id`
- `model_name`
- `fallback_models`
- `status`
- `created_at`
- `updated_at`

`agent_role` must use the existing `AgentRole` values:

- `planner`
- `frontend`
- `backend`
- `reviewer`
- `debugger`
- `security`
- `product`
- `documentation`

Supported `status` values:

- `active`
- `disabled`

There can be at most one active route per `workspace_id` and `agent_role`.

`credential_id` is optional. Fake routes should not require a credential. Non-fake provider routes may be created without a credential for draft configuration, but route resolution must mark them unavailable until they reference an active credential.

### ResolvedModelRoute

`ResolvedModelRoute` is a response shape, not a database table.

Fields:

- `agent_role`
- `provider_name`
- `provider_type`
- `model_name`
- `fallback_models`
- `credential_required`
- `credential_available`
- `is_available`
- `resolution_source`
- `route_id`

`resolution_source` values:

- `configured`
- `fallback_fake`

When no active configured route exists for the requested role, the resolver returns a deterministic fake route:

```json
{
  "agent_role": "planner",
  "provider_name": "fake",
  "provider_type": "fake",
  "model_name": "fake-planner",
  "fallback_models": [],
  "credential_required": false,
  "credential_available": false,
  "is_available": true,
  "resolution_source": "fallback_fake",
  "route_id": null
}
```

### UsageLedgerEntry

`UsageLedgerEntry` is an append-only accounting record for model usage.

Required fields:

- `id`
- `workspace_id`
- `organization_id`
- `user_id`
- `project_id`
- `task_id`
- `planner_run_id`
- `usage_type`
- `provider_name`
- `model_name`
- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `unit_price_cents`
- `amount_cents`
- `raw_usage_json`
- `created_at`

Supported `usage_type` values in Phase 2:

- `model_tokens`

`total_tokens` is derived from prompt and completion tokens by the service layer before persistence. Phase 2 does not calculate real provider cost. `unit_price_cents` and `amount_cents` can be `0`, but the schema exists so billing can later consume the ledger without redesign.

There are no update or delete endpoints for usage ledger entries.

## API Design

### List Model Providers

```text
GET /model-providers
```

Returns active and disabled providers ordered by `created_at`.

### Create Model Provider

```text
POST /model-providers
```

Request:

```json
{
  "name": "deepseek-dev",
  "provider_type": "deepseek",
  "base_url": "https://api.deepseek.com",
  "default_headers": {
    "X-Client": "ai-scdc-dev"
  }
}
```

Response:

```json
{
  "id": "model_provider_abc",
  "workspace_id": "dev_workspace",
  "name": "deepseek-dev",
  "provider_type": "deepseek",
  "base_url": "https://api.deepseek.com",
  "default_headers": {
    "X-Client": "ai-scdc-dev"
  },
  "status": "active",
  "created_at": "2026-05-30T00:00:00Z",
  "updated_at": "2026-05-30T00:00:00Z"
}
```

Behavior:

- Returns 409 if another provider in the workspace already uses the same name.
- Returns 422 for unsupported provider types.
- Rejects header keys that look secret-bearing, such as `Authorization`, `X-API-Key`, or `Api-Key`.

### List Model Credentials

```text
GET /model-credentials
```

Returns credential metadata only.

Response item:

```json
{
  "id": "model_credential_abc",
  "workspace_id": "dev_workspace",
  "provider_id": "model_provider_abc",
  "display_name": "Personal DeepSeek key",
  "secret_last4": "1234",
  "status": "active",
  "created_at": "2026-05-30T00:00:00Z",
  "updated_at": "2026-05-30T00:00:00Z"
}
```

### Create Model Credential

```text
POST /model-credentials
```

Request:

```json
{
  "provider_id": "model_provider_abc",
  "display_name": "Personal DeepSeek key",
  "secret_value": "sk-example1234"
}
```

Behavior:

- Returns 404 if the provider does not exist.
- Returns 400 if the provider is disabled.
- Stores `secret_last4 = "1234"`.
- Stores `encrypted_secret` through `SecretVault`.
- Does not return `secret_value`.
- Does not return `encrypted_secret`.

### Delete Model Credential

```text
DELETE /model-credentials/{credential_id}
```

Behavior:

- Returns 404 if the credential does not exist.
- Soft-deletes the credential.
- Returns a credential read response with `status = deleted`.
- Does not delete or mutate routes that reference the credential.

### List Model Routes

```text
GET /model-routes
```

Returns routes ordered by `agent_role` and `created_at`.

### Create Model Route

```text
POST /model-routes
```

Request:

```json
{
  "agent_role": "planner",
  "provider_id": "model_provider_abc",
  "credential_id": "model_credential_abc",
  "model_name": "deepseek-chat",
  "fallback_models": ["deepseek-reasoner"]
}
```

Behavior:

- Returns 404 if the provider does not exist.
- Returns 404 if `credential_id` is provided and does not exist.
- Returns 400 if `credential_id` belongs to a different provider.
- Returns 409 if an active route already exists for the same workspace and `agent_role`.
- Returns 422 for unsupported agent roles.

### Update Model Route

```text
PATCH /model-routes/{route_id}
```

Request:

```json
{
  "credential_id": "model_credential_def",
  "model_name": "deepseek-reasoner",
  "fallback_models": ["deepseek-chat"],
  "status": "active"
}
```

Behavior:

- Returns 404 if the route does not exist.
- Returns 404 if updated provider or credential references do not exist.
- Returns 400 if the route would reference a credential for a different provider.
- Returns 409 if activating the route would create a second active route for the same agent role.

### Resolve Model Route

```text
GET /model-routes/resolve?agent_role=planner
```

Behavior:

- Returns the active configured route for the role when one exists.
- Returns a deterministic fake route when no active configured route exists.
- Marks non-fake routes unavailable when no active credential is attached.
- Does not call an external provider.

### List Usage Ledger Entries

```text
GET /usage-ledger
GET /usage-ledger?project_id=project_abc
GET /usage-ledger?planner_run_id=planner_run_abc
GET /usage-ledger?task_id=task_abc
```

Returns usage entries ordered by `created_at`.

### Append Usage Ledger Entry

```text
POST /usage-ledger
```

Request:

```json
{
  "project_id": "project_abc",
  "planner_run_id": "planner_run_abc",
  "task_id": null,
  "usage_type": "model_tokens",
  "provider_name": "deepseek-dev",
  "model_name": "deepseek-chat",
  "prompt_tokens": 1200,
  "completion_tokens": 300,
  "unit_price_cents": 0,
  "amount_cents": 0,
  "raw_usage_json": {
    "source": "manual_phase_2_test"
  }
}
```

Response:

```json
{
  "id": "usage_abc",
  "workspace_id": "dev_workspace",
  "organization_id": "dev_organization",
  "user_id": "dev_user",
  "project_id": "project_abc",
  "planner_run_id": "planner_run_abc",
  "task_id": null,
  "usage_type": "model_tokens",
  "provider_name": "deepseek-dev",
  "model_name": "deepseek-chat",
  "prompt_tokens": 1200,
  "completion_tokens": 300,
  "total_tokens": 1500,
  "unit_price_cents": 0,
  "amount_cents": 0,
  "raw_usage_json": {
    "source": "manual_phase_2_test"
  },
  "created_at": "2026-05-30T00:00:00Z"
}
```

Behavior:

- Returns 404 if `project_id`, `planner_run_id`, or `task_id` references a missing resource.
- Returns 400 if `planner_run_id` or `task_id` belongs to a different project.
- Computes `total_tokens = prompt_tokens + completion_tokens`.
- Rejects negative token or price values.

## Service Layer Design

### `model_settings.py`

This service owns provider, credential, route, and route resolution behavior.

Responsibilities:

- Create providers.
- List providers.
- Validate non-secret provider headers.
- Create credentials through `SecretVault`.
- List credential read models without secret fields.
- Soft-delete credentials.
- Create routes.
- Update routes.
- List routes.
- Resolve routes by `AgentRole`.

The service should return Pydantic response models, following the existing `repository.py` pattern for planner read models.

### `secret_vault.py`

This module provides a development-only secret storage boundary.

Public interface:

```python
class SecretVault(Protocol):
    def seal(self, secret_value: str) -> SealedSecret:
        ...


class SealedSecret(BaseModel):
    encrypted_secret: str
    secret_last4: str
```

`DevSecretVault` should produce a non-reversible placeholder suitable for tests, such as:

```text
dev-vault:v1:<sha256 digest>
```

This is not production encryption. It exists to prevent accidental plaintext storage and to preserve the interface where a real KMS or OS keychain can later be plugged in.

### `usage_ledger.py`

This service owns usage entry creation and filtered listing.

Responsibilities:

- Validate optional project, task, and planner-run references.
- Compute `total_tokens`.
- Persist append-only `UsageLedgerEntry` records.
- List entries filtered by project, planner run, or task.

It must not expose update or delete helpers.

### `services/llm-gateway`

The gateway package should remain network-free in Phase 2. It can expand its Pydantic models to align with the backend concepts:

- Provider type enum.
- Route resolution shape.
- Credential reference shape without secret material.
- Usage record with `total_tokens`.

No provider SDK or HTTP client should be introduced in Phase 2.

## Phase 1 Integration

The existing planner approval loop continues to use `FakePlanner`.

Phase 2 must not make planner-run creation depend on route resolution. Planner-run creation must keep working without any configured route, and `GET /model-routes/resolve?agent_role=planner` must return the fake fallback route. Storing resolved route metadata on `PlannerRun` is deferred until the real model-backed planner phase.

No Phase 2 code may make planner-run creation depend on a configured credential.

## Error Handling

The API must handle:

- Duplicate provider name: 409.
- Unsupported provider type: 422.
- Secret-bearing default provider header: 400.
- Missing provider: 404.
- Disabled provider during credential creation: 400.
- Missing credential: 404.
- Credential for wrong provider: 400.
- Duplicate active route for role: 409.
- Invalid agent role: 422.
- Missing project/task/planner run in usage append: 404.
- Cross-project task or planner run in usage append: 400.
- Negative usage quantities or amounts: 422.

Secret handling failures should return a generic 400 or 500 response without echoing submitted secrets.

## Testing Strategy

### API Tests

Add tests for:

- Creating and listing a model provider.
- Duplicate provider names return 409.
- Provider creation rejects secret-bearing default headers.
- Creating a credential returns only metadata.
- Created credential does not persist raw plaintext.
- Credential response schemas do not include `secret_value` or `encrypted_secret`.
- Deleting a credential soft-deletes it.
- Creating a route stores provider, credential, model, and fallback model metadata.
- Creating a second active route for the same role returns 409.
- Updating a route changes model and fallback model metadata.
- Route resolution returns configured route when available.
- Route resolution returns fake fallback when no route is configured.
- Route resolution marks a non-fake route unavailable when its credential is deleted.
- Appending usage computes `total_tokens`.
- Usage listing filters by project, planner run, and task.
- Usage append rejects cross-project references.
- Usage ledger has no update or delete endpoints in OpenAPI.

### Service Tests

Add tests for:

- `DevSecretVault` never returns plaintext as `encrypted_secret`.
- `DevSecretVault` returns the correct `secret_last4`.
- Route resolution is deterministic for the same agent role and empty route table.
- Usage service computes `total_tokens` from prompt and completion tokens.

### Gateway Tests

Add tests for:

- Gateway usage records expose `total_tokens`.
- Route models serialize without credential secrets.
- Provider config models support OpenAI-compatible and DeepSeek provider metadata without network calls.

### Verification Commands

Phase 2 is complete only when these pass:

```bash
git diff --check
```

## Success Criteria

Phase 2 is complete when:

- A developer can create provider metadata through the API.
- A developer can create BYOK credential metadata without exposing raw or encrypted secrets through responses.
- A developer can configure and update role-based model routes.
- `planner` route resolution returns a usable fake route with no configuration.
- Configured non-fake route resolution reports whether a credential is active and available.
- Usage ledger entries can be appended and queried by project, planner run, and task.
- Usage ledger has no update or delete API.
- Phase 1 planner approval flow still works unchanged.
- The repository passes the agreed test and typecheck commands.

## Out-of-Scope Deferral Notes

The next model-related phase can replace the fake planner with a model-backed planner by using route resolution and an OpenAI-compatible adapter behind the existing `PlannerService` interface.

The future production secret vault can replace `DevSecretVault` without changing API response models.

The future billing phase can consume `UsageLedgerEntry` records to mutate credit wallets, invoices, and subscription usage.
