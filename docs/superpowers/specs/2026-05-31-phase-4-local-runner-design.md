# AI Company Console Phase 4 Local Runner Design

## Goal

Build the first local execution path for **AI Software Company Desktop Console** after a planner run has been approved.

Phase 4 turns approved task records into reviewable local patch artifacts:

```text
approved planner drafts
  -> normal tasks
  -> repository registration
  -> local runner starts task
  -> git worktree is created under .worktrees
  -> runner makes a bounded worktree change
  -> git diff is captured as a patch artifact
  -> task transitions to PATCH_READY
  -> user reviews the diff
```

The phase is intentionally local and review-first. It does not auto-merge, does not push branches, and does not run a reviewer/debugger loop.

## Confirmed Direction

- Phase 4 is the Local Runner vertical slice.
- The local runner works against repositories already present on the developer machine.
- Work happens in git worktrees, not in the source checkout.
- Generated patches are artifacts for human review.
- Task state changes continue to use the API task state machine and task events.
- The implementation remains deterministic enough for automated tests.
- The first runner may use a bounded deterministic patch strategy; general LLM coding is deferred.

## Scope

Phase 4 includes:

- Storing task execution inputs that were missing from Phase 1 approval copying:
  - `allowed_paths`
  - `required_tests`
- A repository registry for local development repositories.
- Local run records tied to tasks and repositories.
- Patch artifact records containing summary, files changed, tests run, test result, risks, and captured diff text or a local artifact path.
- A worker-side local runner that:
  - validates repository paths,
  - checks git availability,
  - creates a worktree under the repo-local `.worktrees` directory,
  - applies a constrained demo patch in the worktree,
  - captures `git diff`,
  - returns a `patch-result`-compatible payload.
- API endpoints to start and inspect local runs.
- Desktop controls to start a local run for a task and inspect the resulting patch metadata.
- Tests covering repository validation, worktree creation, task transitions, patch artifact creation, and desktop state updates.

Phase 4 excludes:

- Automatic merge, push, PR creation, or branch publication.
- Cloud sandbox workers.
- Arbitrary shell commands generated by a model.
- Long-running background job queues.
- Real reviewer, debugger, or repair loops.
- Production repository credentials.
- Billing changes.

## Existing Starting Points

The repo already has:

- `Task.repo_id`, `Task.branch_name`, and `Task.worktree_ref` fields.
- A task state machine with `ASSIGNED`, `IN_PROGRESS`, `PATCH_READY`, `REVIEWING`, and later review states.
- `TaskEvent` audit records.
- `apps/worker` with a deterministic simulator and API state-machine validation.
- Agent protocol `patch-result.schema.json`.
- Planner drafts with `allowed_paths` and `required_tests`, but approved `Task` rows do not yet persist those fields.

## Architecture

### Repository Registry

The API owns repository records for local development:

- `Repository.id`
- `project_id`
- `name`
- `local_path`
- `default_branch`
- `status`
- `created_at`
- `updated_at`

The first version is development-only and stores a local filesystem path. The API validates that the path exists and contains a git repository before accepting it.

### Task Execution Inputs

Approved tasks must preserve planner execution constraints:

- `allowed_paths` is copied from each planner draft to the created task.
- `required_tests` is copied from each planner draft to the created task.

These fields let the runner enforce the planner boundary and report which test commands were expected, even before Phase 5 adds automated test execution.

### Local Run Records

The API tracks each run:

- `LocalTaskRun.id`
- `task_id`
- `project_id`
- `repo_id`
- `status`
- `runner_kind`
- `base_branch`
- `base_sha`
- `head_sha`
- `worktree_path`
- `patch_artifact_id`
- `failure_reason`
- timestamps

Initial statuses:

- `queued`
- `running`
- `patch_ready`
- `failed`

The first implementation can execute synchronously inside the API request for local development, but the data model should not assume that forever.

### Patch Artifacts

Patch artifacts are the review boundary:

- `PatchArtifact.id`
- `task_id`
- `local_run_id`
- `summary`
- `files_changed`
- `tests_run`
- `test_result`
- `risks`
- `diff_text`
- `created_at`

The payload should satisfy the existing agent protocol `patch-result` shape. `diff_text` is API-specific review data and can remain outside the shared schema.

### Worker Local Runner

`apps/worker` gains a real local runner service separate from the simulator.

Responsibilities:

- Resolve and validate source repository path.
- Read current branch and base commit.
- Create a task-specific worktree under `.worktrees/<task_id>-<run_id>`.
- Apply a small bounded patch in the worktree.
- Validate changed files against `allowed_paths`.
- Capture changed files and `git diff --no-ext-diff`.
- Report `PatchResult`.

The first patch strategy should be deterministic and safe:

- Prefer a text file explicitly allowed by the task, such as `README.md`.
- Otherwise create a local runner note only under an allowed directory or glob.
- Never edit files outside the worktree.
- Never stage, commit, push, or merge.

### API Flow

1. User registers a repository for a project.
2. User approves a planner run, creating tasks with `allowed_paths` and `required_tests`.
3. User starts a local run for one task.
4. API validates task and repository ownership.
5. API creates a `LocalTaskRun` and transitions task through:
   - `ASSIGNED`
   - `IN_PROGRESS`
6. API invokes the worker local runner in-process for Phase 4.
7. Worker returns a patch result and diff.
8. API persists `PatchArtifact`.
9. API updates task:
   - `repo_id`
   - `branch_name`
   - `worktree_ref`
   - `PATCH_READY`
10. API writes task events for local run lifecycle and patch artifact creation.
11. Desktop shows patch-ready metadata for review.

## Safety Boundaries

- Use subprocess argument arrays, not string-built shell commands.
- Resolve absolute paths before git operations.
- Refuse repositories outside explicitly registered local paths.
- Refuse worktree paths outside the repository `.worktrees` directory.
- Refuse dirty source repository state unless the implementation can prove it does not matter.
- Refuse patches that change files outside `allowed_paths`.
- Keep API keys and model credentials out of runner logs and patch artifacts.
- Do not delete worktrees recursively unless the resolved path is verified to be inside the intended `.worktrees` root.
- Do not auto-merge or commit generated changes.

## Desktop Behavior

The desktop should remain an operational console, not a code editor.

Phase 4 desktop additions:

- Repository selection or current repository display for API-backed mode.
- A run-local action on tasks that are eligible for local execution.
- Local run status and patch summary in the task panel.
- Files changed and tests-run metadata.
- A diff preview or link to fetch patch artifact details.

The UI should keep approval boundaries clear: a patch being ready does not mean it is merged.

## Testing Strategy

Worker tests:

- local runner rejects non-git paths,
- local runner creates worktree under `.worktrees`,
- local runner captures changed files and diff,
- local runner refuses changes outside `allowed_paths`,
- local runner does not modify the source checkout.

API tests:

- repository endpoints validate and return repository metadata,
- approving planner drafts copies `allowed_paths` and `required_tests` to tasks,
- starting a local run creates `LocalTaskRun` and `PatchArtifact`,
- task transitions reach `PATCH_READY`,
- task events record local runner lifecycle,
- cross-project repository/task usage is rejected,
- failed runner records failure reason and leaves an auditable event.

Desktop tests:

- API client supports repository and local-run endpoints,
- task board can trigger a local run,
- patch-ready metadata renders without breaking existing planner approval flow.

## Success Criteria

Phase 4 is complete when:

- A developer can register a local git repository for a project.
- An approved task can be run through the local runner.
- The local runner creates a git worktree and produces a captured diff.
- The API persists a patch artifact and task events.
- The task reaches `PATCH_READY` without merge, push, or PR creation.
- Desktop users can start the run and inspect patch metadata.
- Tests pass with no external network dependency.
- Full verification passes:

```bash
pnpm test
pnpm typecheck
pytest apps/api/tests apps/worker/tests services/llm-gateway/tests -v
git diff --check
```

## Future Deferrals

- LLM-powered code editing.
- Automated test execution from `required_tests`.
- Reviewer loop and debugger loop.
- Worktree cleanup policies and retention UI.
- GitHub/GitLab branch push and PR creation.
- Cloud sandbox execution.
- Organization-level repository credentials and RBAC.
